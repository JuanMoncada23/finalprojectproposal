The front end of the WebApp can currently:

  • Expire tasks upon reaching the deadline
  • Mark tasks as finished, halting the countdown
  • Load tasks from local JSON objects

  • Run within a docker container

The storage solution can currently:

  • Store data sent to it by the logic-tier

End-User Description

The application allows users to create and set timers for a list of tasks to be completed by a certain time. The user receives periodic reminders in the browser about active timers as well as “constructive” criticism regarding how close their tasks come to expiring.

Priority Features

The features required to make Deadline Web Service usable are:

  1. Create and edit a list of tasks and their deadlines
  2. Attach a countdown timer to each task
  3. Store tasks persistently for users
  4. Expire tasks once the deadline is passed, sounding an alarm to the user
  5. Provide periodic reminders about active tasks based on remaining time
  6. Provide motivation to complete tasks before the deadline
  7. Mark tasks as finished, halting the countdown

Additional Desired Features

If there were extra time, we would like to add:

  • Daily emails to users about their active tasks
  • Marking tasks with a higher or lower priority
  • Presenting average time taken per task and tracking improvement
  • Color-code tasks based upon remaining time

Technical Requirements

Building the priority features will require:

  • HTML for webpage content
  • CSS for webpage styling
  • JavaScript / React for webpage interactions
  • Go for the logic tier
  • A database storage solution (MongoDB Atlas) for persistent storage
  • Microsoft Azure for app hosting and data storage

Demonstrating Learning From Outside the Class

Using Asynchronous JavaScript Fetch Requests

Our time in class was more focused on single-page apps that did not have to perform any requests to other resources in order to function. Deadline Web Services does need to do this. Specifically, the front end of the WebApp (written in React) interacts with the storage solution (MongoDB Atlas) by performing asynchronous JavaScript fetch requests to the logic-tier (written in Golang), which performs the database query and returns some data. As an additional challenge, the logic-tier implements this interaction by exposing RESTful endpoint APIs. This means that the front end needs to POST JSON data within the request, instead of a simple GET request. The solution we implmented is to use JavaScript fetch() requests with the async / await syntax, which forces the WebApp to wait for the request to finish before continuing. This prevents data from being POSTed to the logic-tier out of order, which would cause issues with the database queries.
